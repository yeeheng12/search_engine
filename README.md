# Search Engine — COMP3011 Coursework 2

A Python-based search engine that crawls quotes.toscrape.com, builds an inverted index, and provides a command-line interface to search pages by keyword.

---

## Project Overview

This project was built for COMP3011 Web Services and Web Data at the University of Leeds. The goal was to implement the core components of a search engine from scratch: a web crawler, an inverted indexer, and a query interface.

The system follows a clear pipeline:

```
crawl pages
     ↓
parse HTML (BeautifulSoup)
     ↓
tokenise text (lowercase, alphanumeric)
     ↓
build inverted index (word -> doc_id -> frequency + positions)
     ↓
save index to JSON
     ↓
load index from JSON
     ↓
search via CLI (print / find)
```

## Usage

Start the search engine shell:

```
cd src
python main.py
```

### Commands

**build** — crawls the full website, builds the inverted index, and saves it to data/index.json

```
> build
```

**load** — loads a previously saved index from data/index.json without re-crawling

```
> load
```

**print** — prints the posting list for a single word, showing each page it appears in, its frequency, and token positions

```
> print nonsense
```

**find** — returns all pages that contain every word in the query, ranked by total frequency

```
> find indifference
> find good friends
```

**help** — lists all available commands

```
> help
```

**quit** — exits the program

```
> quit
```

## Data Structures

### Inverted Index

Each word maps to a posting list. Each posting contains the document ID, how often the word appears, and which token positions it occupies.

```
{
  "good": {
    "1": {
      "frequency": 2,
      "positions": [5, 18]
    }
  }
}
```

Positions are stored so that phrase proximity could be supported in future. This follows the posting list model described in the indexing lecture, where each term has an inverted list and each entry points to a document.

### Document Table

Document IDs are mapped back to page metadata in a separate table. This keeps the index compact and avoids repeating full URLs across every posting.

```
{
  "1": {
    "url": "https://quotes.toscrape.com/page/1/",
    "title": "Quotes to Scrape",
    "word_count": 140
  }
}
```

### Full Index File Structure

The entire index is stored in a single JSON file under data/index.json.

```
{
  "metadata": {
    "created_at": "2026-05-08T12:00:00",
    "page_count": 52,
    "term_count": 2145,
    "politeness_delay": 6
  },
  "documents": { ... },
  "index": { ... }
}
```

## Design Decisions

**BFS crawler** — the crawler uses a queue and visited set to traverse the site breadth-first. This mirrors the frontier model from the crawling lecture, which describes crawling as graph traversal. A queue was chosen over recursion because it is easier to test, easier to control, and avoids stack overflow on larger sites.

**6-second politeness delay** — a sleep of 6 seconds is called between every request, as required by the coursework brief. The minimum is enforced in code even if a lower value is passed, so it cannot be accidentally bypassed.

**Document ID mapping** — the index stores short integer IDs rather than full URLs. A separate document table maps each ID back to its URL and metadata. This follows the lecture's explanation of why document numbers are stored instead of full identifiers for efficiency.

**Conjunctive search** — the find command uses AND logic. All query words must appear in a page for it to be returned. This matches the coursework example where find good friends returns only pages containing both words, not pages that contain just one of them.

**Frequency-based ranking** — results are scored by the total frequency of all query terms in each matching document. Pages where the query words appear more often are ranked higher. This is a simple but explainable relevance signal.

**Consistent tokenisation** — the same tokenisation function is used for both indexing and query processing. Both convert to lowercase and extract only alphanumeric tokens using a regex. Without this consistency, a word stored as good at index time could fail to match a query for Good or GOOD.

**JSON storage** — the index is saved to a single JSON file. It is human-readable, straightforward to inspect during the video demonstration, and sufficient for the scale of this project.

## Complexity Analysis

| Operation | Complexity | Explanation |
|-----------|-----------|-------------|
| Index build | O(N) | N = total tokens across all pages |
| Single word lookup | O(1) | Dictionary hash lookup |
| Multi-word find | O(Q x D) | Q = query terms, D = posting list size |
| Storage | O(T + P) | T = unique terms, P = total postings |

The inverted index structure was chosen specifically because single-word
lookup is O(1) average case — significantly faster than scanning every
document for each query.


## Testing

Run all tests:

```
pytest tests/ -v
```

Run with coverage report:

```
pytest tests/ --cov=src --cov-report=term-missing
```

### Latest Test Results

65 tests passed in 1.45s
Total coverage: 89%
Coverage threshold enforced: 85% minimum via pytest.ini

### What Each Test File Covers

**test_indexer.py** — tokeniser lowercase conversion, punctuation removal, number handling, frequency counting, position recording, multi-page indexing, empty page handling, build reset behaviour, and save/load roundtrip.

**test_search.py** — single-word find, multi-word conjunctive intersection, case-insensitive query handling, frequency-based ranking, empty query handling, whitespace-only queries, missing word behaviour, and return type verification.

**test_crawler.py** — BFS link extraction, internal versus external URL filtering, login page exclusion, relative URL conversion to absolute, politeness delay verification, timeout handling, HTTP error handling, and duplicate URL prevention. All crawler tests use mocked HTTP responses and make no real network requests.
